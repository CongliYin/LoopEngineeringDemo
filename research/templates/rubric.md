# Research Rubric

报告必须满足：

1. 包含 $required_sections。
2. 至少引用 $min_sources 个来源。
3. 至少 $min_source_year_ratio 的来源必须来自 $source_year 年，按 `sources.json` 中每个来源的 `date` 字段判断。
4. `date` 必须使用 `YYYY-MM-DD` 或 `YYYY` 开头。
5. 来源 ID 必须从 `S1` 开始连续编号，URL 不能重复，字段必须包含 id、title、url、publisher、date、summary、relevance。
6. 每个正文章节至少 $min_section_chars 个非空白字符，且至少包含一个有效来源引用。
7. 每个来源都必须在正文中被引用，不能只出现在参考来源列表。
8. 参考来源部分必须列出每个来源的 ID 和 URL。
9. 避免无来源的大词和空泛判断。
10. `技术全景与发展脉络` 必须讲清楚 2022-2026 的技术演进，不允许只列 2026 年论文。
11. `关键技术路线详解` 是全文最重要的部分，必须明显长于其他单个章节；如果它只占少量篇幅，或只是列表式概述，不能通过。
12. `关键技术路线详解` 至少覆盖 6 条路线。每条路线必须使用三级标题，并包含：路线定义、要解决的问题、代表来源深读、核心机制、工程实现方式、适用场景、局限和落地建议。
13. 代表来源不能只一句话带过。对每条路线至少选择 2 个代表来源做深读：说明来源具体做了什么、任务/环境/数据是什么、方法结构是什么、如何评估、结论边界是什么、工程上可以借鉴什么。
14. `关键技术路线详解` 必须包含 Mermaid 技术路线图，以及一张路线总览表。表格只能用于导航，不能替代正文展开。
15. 报告排版必须适合技术分享阅读：使用清晰的三级标题、短段落、表格、列表、图示、代码块；避免连续多段大段文字堆叠。
16. `工程实现与代码设计` 必须说明 loop、验证器、状态、权限和人工升级，并包含具体代码片段和 Mermaid 代码流程图。
17. `落地选型与使用建议` 必须说明不同团队/场景应该优先采用哪些路线、前置条件、成本、收益和不适用场景。
18. `评估方法与实验设计` 必须给出可执行 benchmark、指标、基线、ablation、统计方法、长期演化归因、安全回归和线上门禁阈值。
19. 不要求固定运行 2 轮；只要 deterministic validator 和 Codex validator agent 同时通过且 agent 没有修改建议，外部 loop 就可以停止。
